Porting SoX's `spectrogram` effect to Rust (WAV input only). Source of truth: `sox-code/` (version **14.4.3git** per configure.ac). Installed SoX binary is **14.4.2 (32-bit, x87)** at `C:\Program Files (x86)\sox-14-4-2\sox.exe`.

**Modules (all written, builds clean 0 warnings, 10 unit tests pass):** `fft.rs` (**rustfft** — single `Dft` type, `accumulate_power`; replaced hand-ported Ooura rdft AND slow non-p2 DFT), `audio.rs` (**symphonia** — replaced the hand-rolled `wav.rs`; decodes all symphonia formats, bridges every sample through SoX's `signed/unsigned/float_to_sample`), `window.rs`, `timeparse.rs` (-d/-S), `spectrogram.rs` (Config/geometry/flow/drain), `render.rs` (palette/font/axes/PNG), `tables.rs` (FIXED_FONT_ZLIB + ALT_PALETTE), `main.rs` (clap CLI, humanized long flags).

**Deps:** clap (derive), png, flate2, rustfft, symphonia (features=["all"]) — user adds via `cargo add` themselves (see [[deps-and-edition]]).

**Ongoing effort: replacing C-style ported code with proper Rust libraries.**
- FFT → rustfft (2026-06-08): output *identical* to prior Ooura build (0 px diff), still 0-diff vs SoX in raw mode incl. non-p2 sizes. rustfft ~1e-13 diffs are below the colour-quantization step.
- WAV → symphonia 0.6 (2026-06-08): bit-exact preserved for ALL lossless paths — verified 0 raw-mode diffs vs SoX for 8/16/24/32-bit + 32f WAV AND FLAC. Adds MP3/OGG/AAC/ALAC/etc. (MP3 couldn't be round-trip tested: SoX 14.4.2 has no LAME encoder). Symphonia 0.6 API notes: `get_probe().probe(&hint, mss, FormatOptions, MetadataOptions)` (by value) → `Box<dyn FormatReader>`; `next_packet()->Result<Option<Packet>>`; buffers are `GenericAudioBufferRef` (variants U8..F64); planar access via the `Audio` trait `.plane(c)`; `i24/u24` are tuple structs (`.0`); decoder via `get_codecs().make_audio_decoder(params.audio(), &AudioDecoderOptions::default())`.

**Migrated floem 0.2.0 (crates.io) → git (2026-06-09):** Reason: 0.2.0 is ~2yr stale and has startup-flicker bugs (window resize-jump + white flash) that are fixed upstream. User added `floem = { git="https://github.com/lapce/floem", rev="bcafaaad3fbf47091e28a64934295194d8e14463", version="0.2.0" }`. Migration was SMALLER than feared (most paths compatible). Changes made to gui.rs: (1) events are now TYPED listeners — `floem::event::listener::{FileDragDrop, WindowResized}` + `.on_event_stop/on_event_cont(listener::X, |cx: &mut EventCx, data: &L::EventData| ...)`. Drag-drop: `Event::DroppedFile` → `listener::FileDragDrop` whose data is `FileDragDropped { paths: Rc<[PathBuf]>, position }` (use `ev.paths.first()`). (2) NO per-view `.on_resize` anymore — only window-level `listener::WindowResized` (payload `Size`); derive centre area = window − gutters via `centre_area(w,h)`; init `area` from configured window size. (3) `peniko::Color` is now `AlphaColor` → `Color::rgb8` → `Color::from_rgb8`. (4) Fonts: cosmic-text → **Parley** (`floem::text::FONT_CONTEXT: Mutex<parley::FontContext>`); old `FONT_SYSTEM` gone. Currently RELY ON SYSTEM-INSTALLED JetBrains Mono (dropped `load_fonts()`); embedding via parley/fontique `register_fonts` is TODO if portability needed (asset still at assets/JetBrainsMono.ttf, now unused). (5) `.px()` deprecated → `.pt()`. (6) view free-fns deprecated → constructors: `create_rw_signal`→`RwSignal::new`, `text`→`Label::new`, `label`→`Label::derived`, `h_stack`→`Stack::horizontal`, `v_stack`→`Stack::vertical`, `v_stack_from_iter`→`Stack::vertical_from_iter`, `stack_from_iter`→`Stack::from_iter`, `empty`→`Empty::new` (CAREFUL: blind sed broke `is_empty()`→`is_Empty::new()`). Build clean 0/0. Flicker FIXED (user confirmed UX issues gone). BUT first attempt showed a BLANK window: in git floem the view tree + its signals MUST be built INSIDE the window/app-view closure (floem's runtime scope), not before `Application::new()` like floem 0.2 allowed — building outside → dead scope → nothing mounts. FIX: moved all construction into `fn build_ui(initial, fft_size) -> impl IntoView` called from `.window(move |_| build_ui(...), cfg)`. (Examples create signals inside the view fn, e.g. `floem::launch(counter_view)`.) NOTE there's now a `views::canvas` view — could draw the spectrogram directly (no PNG round-trip) if desired later.

**GUI added (floem 0.2.0, 2026-06-08):** `src/gui.rs`, Spek-like drag-and-drop viewer. `main.rs`: `input` is now `Option<PathBuf>` + `--gui` flag; no input or `--gui` → `gui::run`. floem 0.2 notes: entry `Application::new().window(|_id| view, Some(WindowConfig::default().size(Size::new()).title()))`.run(); reactive signals via `floem::reactive` (`create_rw_signal`, `RwSignal: Copy`, traits `SignalGet`/`SignalUpdate`); `views::{dyn_view, img, label, container, Decorators}`; drag-drop = `.on_event_stop(EventListener::DroppedFile, |e| if let Event::DroppedFile(ev)=e {ev.path})`; debounce via `floem::action::exec_after`. KEY GOTCHA: floem 0.2 `img` reads its bytes closure ONCE (not reactive) and lays out at the image's NATIVE pixel size (ObjectFit defined but unused) — so we re-render the spectrogram to the window size on `.on_resize` (debounced 150ms) and drive `img` from a `dyn_view` over an `RwSignal<Vec<u8>>` of PNG bytes (reuses `render_png`, no render_rgba needed). `gen` is a reserved keyword in edition 2024 (don't name vars `gen`).

**GUI perf fix (2026-06-08):** first version re-rendered the spectrogram at the FULL window size on every resize. On a 4K/HiDPI display `on_resize` reports PHYSICAL px (saw 3824 wide), so it rendered multi-thousand-px images and floem decoded+SHA256-hashed+GPU-uploaded them on the UI thread → ~10s freeze on a normal FLAC + unresponsive resize. Fix: floem's `draw_img` GPU-scales the texture to the view rect (`vger.render_image(w,h)` with native-size texture), so style the `img` `size_full` and render at a SMALL texture matching the window aspect (`MAX_RENDER=1600` longest side, aspect-preserved via `render_target()`). Resize is now instant (GPU scaling during drag) with a cheap (~100-150ms, ≤1600px) debounced re-render on settle to correct aspect. Texture stays tiny so floem decode/upload is cheap and GPU-limit-safe. NOTE: load/process still on UI thread (~0.5-1s one-time on drop) — could background-thread via `floem::ext_event` if that hitch matters. Couldn't visually verify fill/quality headlessly.

**GUI perf rework #2 (2026-06-08) — user said cap-version was STILL unacceptably slow, and EXPLICITLY lifted the SoX-visual-parity requirement for the GUI ("use other drawing mechanisms if it helps efficiency").** Root issue was architectural: the spectrogram FFT was recomputed on EVERY resize. New design (current): (1) compute the spectrogram ONCE per file; (2) on a BACKGROUND THREAD via `floem::ext_event::create_ext_action(Scope::current(), move |res| ...)` + `std::thread::spawn` (the returned sender is Send/FnOnce — moved into the thread, called once with the result) so drop never freezes the UI; (3) render a plain HEATMAP straight to RGBA→PNG via new `render::heatmap_png` (no axes/legend/font — NOT SoX-parity; channels stacked, high-freq at top); (4) display `img(size_full)` so floem GPU-scales the texture to the window → resize is FREE, no recompute/re-render. Fixed render resolution `RENDER_COLS=1600, RENDER_HEIGHT=900` (GPU scales to window; image is stretched to fill, aspect not preserved — acceptable for a heatmap). Removed all the on_resize/debounce/exec_after/view_size machinery. CONSEQUENCE: the GUI no longer shows axis/time/frequency/dB labels — could re-add later as floem text overlay views (crisp, unaffected by image scaling). Preload (CLI `--gui file`) path is synchronous (before event loop, no Scope yet); only the drop path is threaded. Still couldn't visually verify fill/responsiveness headlessly — needs user confirmation.

**`--fft-size` flag (2026-06-09):** `-F/--fft-size N` (64-16384) sets the FFT/window size directly via `cfg.y_size = N/2 + 1` (since `dft_size = 2*(y_size-1)`). `conflicts_with_all=["height","total_height"]`. Applies to BOTH the CLI render path (build_config) and the GUI (threaded `gui::run(initial, fft_size)` → `analyze`/`build_image`; when set it uses y_size per-channel instead of the default `big_y_size=RENDER_HEIGHT`, so all channels get the same N-pt FFT). Header "N-pt" reflects it. Note: GUI FFT size is fixed at launch (does NOT change on resize — by design; resize only GPU-scales the precomputed texture). Context: "pt" in header = FFT points.

**GUI-by-default dispatch (2026-06-09):** `main.rs` now opens the GUI by default. PNG render (CLI) only when `--output/-o` is given AND `--gui` isn't. `output` is now `Option<String>` (no default filename). **Release size + no console window (2026-06-09):** Added `[profile.release]` to Cargo.toml (NOT deps): `opt-level=3` (kept, for FFT speed), `lto="fat"`, `codegen-units=1`, `panic="abort"`, `strip=true` → 18.63MB → 12.80MB (~31%). Fat-LTO release builds now take ~2-2.5 min. Biggest remaining lever (a dep change, user's call): symphonia `features=["all"]` pulls every codec — narrowing it saves several MB. Console-window fix: `#![cfg_attr(windows, windows_subsystem="windows")]` (no console pops up on GUI launch) + `attach_parent_console()` at main() start (FFI `unsafe extern "system" { fn AttachConsole(u32)->i32 }`, ATTACH_PARENT_PROCESS=0xFFFFFFFF, `#[link(name="kernel32")]`) so CLI run from a terminal still prints (no-op when double-clicked → no console). Edition 2024 requires `unsafe extern`. cargo test unaffected (dev profile unwinds).

**Per-channel rendering (2026-06-09):** Channels are now rendered as SEPARATE images (`render::channel_pngs(spec) -> Vec<Vec<u8>>`, one PNG per channel) and laid out in a `v_stack_from_iter` of `img` views, each `flex_grow(1)` + `min_*(0)`, with `row_gap(CHANNEL_GAP_PX=8.0)` — so the channel gap is real window background, not baked into one stacked texture (user: "gap drawn as part of the spec, split renders per channel"). Old single-stacked `heatmap_png`/`heatmap_rgba` + `render::CHANNEL_GAP` removed. image signal is now `RwSignal<Vec<Vec<u8>>>`. Freq-axis band math switched from native-row ratios to layout-based equal bands: `band_h = (area_h - (n-1)*gap)/n`, `band_top = k*(band_h+gap)`. SpecMeta dropped its `rows` field (now carries `dft_size`).

**White-flash on launch (2026-06-09, NOT fixed — floem limitation):** Window shows OS-white before floem's wgpu renderer finishes async init (it `GpuResources::request`s AFTER creating a visible window). floem 0.2 `WindowBuilder` is created visible with no `with_visible(false)` and `WindowConfig` has no visibility/background option; the per-frame clear color (theme bg / white) is always covered by our opaque root so it's NOT the cause. No clean fix in 0.2 (would need window-start-hidden, i.e. a floem upgrade). Told user it's inherent.

**Layout gotcha #2 — gutters collapsing (2026-06-09):** Axis-label/legend gutters whose content is ALL absolutely-positioned views have ZERO intrinsic size, so flexbox happily shrinks them to 0 at smaller window sizes → the heatmap (flex_grow) expands over them and the absolute labels render on top of / behind the spectrogram ("text behind the spec", labels "vanish" small but fine large). FIX: `flex_shrink(0.0)` on every fixed gutter + spacer (freq/legend/time spacers/title/bottom row), so they reserve their space and the heatmap is the ONLY thing that gives (it has flex_grow + min_width(0)). (Misdiagnosed this as a repaint issue first — it was not.)

**Layout gotcha (2026-06-09):** Don't put `.size_full()` on a flex child that should share space with siblings — `size_full` = width:100% AND height:100%, so the centre claimed 100% width and squeezed the side gutters (freq/legend) to nothing, and the row claimed 100% height squeezing the time axis (symptom: heatmap filled the whole window, no axes). Use `.flex_grow(1.0)` + only the CROSS-axis full: centre = `flex_grow(1).height_full()` (in the horizontal row), row = `flex_grow(1).width_full()` (in the vertical v_stack).

**Two more floem gotchas (2026-06-09):** (a) An absolutely-positioned `text()`/label with NO explicit width/height collapses to 0×0 and is invisible (gridline `empty()` views showed because they had explicit w/h). FIX: give axis labels explicit `.width().height()`. (b) After a worker-thread signal update, `request_all` only repaints the targeted view's chain + ancestors — SIBLING gutters (legend/axes) stayed stale ("legend vanished after drop"). FIX: capture the ROOT view id (`v_stack(...).id()` before `.on_event_stop`) and `root_id.request_all()` so the whole tree repaints.

**CRITICAL floem gotcha (cost a debugging cycle):** `on_resize` fires *during* the layout pass. If you `signal.set` there and that rebuilds text/`label`/`text()` views synchronously, the freshly-created labels hit `compute_layout` before their `layout()` runs, so `Label::text_layout` is still `None` → panic at `label.rs:441` (`text_layout.as_ref().unwrap()`). FIX: defer the signal update out of the layout pass, e.g. `exec_after(Duration::from_millis(0), move |_| area.set(sz))`. (Symptom was: no-arg/placeholder fine, but loading a file panicked.)

**Parallelism (rayon 1.12, 2026-06-08):** `spectrogram::process` now processes channels in parallel via `(0..chans).into_par_iter()`, each task building its own `Dft` (FFT plan) — fully independent, output bit-identical (verified stereo raw = 0 diffs vs SoX). Helps stereo (~2× on the FFT, the dominant cost); mono unaffected. Measured: FLAC decode is cheap (~31ms; symphonia FLAC is fast) so it's NOT worth parallelizing decode — the FFT `process` dominates. Channel result collected into `ChannelOutput { dbfs, max, cols }`; ch0 max used for normalize. POSSIBLE bigger win if asked: column/time-block parallelism (helps mono, scales with cores) but it's a risky refactor of the bit-exact flow/drain state machine.

**Assessment of remaining C-style code (told user 2026-06-08):** `timeparse.rs` is NOT a good library candidate — SoX's `mm:ss`/sample-count/`=+-` anchor grammar matches no popular crate; leave it or do an idiomatic rewrite. `window.rs` must stay (SoX-specific Kaiser-beta poly + Dolph). `render.rs`/`tables.rs` font+palette must stay for pixel fidelity. Next genuine win if asked: error handling `Result<_,String>` → thiserror/anyhow.

**Verification vs installed SoX (test/ has mono.wav, stereo.wav + generators):**
- `-r` raw mode (pure spectrogram data): **bit-exact**, 0/410400 px diff. 36/39 x-values in a sweep are 0-diff.
- Full images: identical except divergences traced to the 14.4.2 binary being a **32-bit x87 build** (80-bit intermediates). Two confirmed signatures, both "true value just below an integer, x87 truncates down, IEEE-754 double rounds up": (1) Z-legend gradient single row, `colour()` `83.999…` vs `84.0` (~14 px); (2) `step_size`/`block_steps` at specific x (e.g. 700: `44100/pps` = 188.99999999999999233 -> x87 188, double 189) which cascades. NOT a port bug — a modern 64-bit SoX build would match rustspeck.
- `-A` alt palette differs because 14.4.2 alt_palette had 169 entries vs 14.4.3git source's 168 (genuine version difference).
- `-n` normalize: 14.4.2 binary lacks it (`invalid option -n`); it's a 14.4.3git feature, implemented per source.

**Canvas heatmap switch (2026-06-09):** GUI heatmap no longer goes through PNG encode→decode→GPU-upload. `render::channel_pngs` → `render::channel_images(spec) -> Vec<(w,h,Vec<u8>)>` (raw RGBA per channel). gui.rs draws via a floem `views::canvas(|cx, size| ...)` that `cx.draw_img(floem_renderer::Img{ img: ImageBrush, hash: &Arc<[u8]> }, Rect)` per channel band (band math same as before: `band_h=(h-(n-1)*gap)/n`). Each channel RGBA wrapped once in an `ImageBrush` (`peniko::{Blob::new(Arc<Vec<u8>>), ImageData{format:Rgba8, alpha_type:Alpha, w,h}, ImageBrush::new(..).with_quality(High)}`) with a stable hash key `(NEXT_GEN.fetch_add, channel_idx)` so the renderer caches/uploads each texture once and GPU-scales on resize. image signal type: `RwSignal<Vec<ChannelTex>>`. NOTE: `gen` is reserved in edition 2024 — name it `gen_id`. Needs `floem_renderer` dep (user added) + `floem` and `floem_renderer` BOTH with `features=["crossbeam"]`.

**CROSSBEAM FEATURE GOTCHA (2026-06-09):** floem's gpu-resource handoff (`window/handle.rs` → `PaintState::new_pending`) picks `crossbeam::channel::Receiver` vs `std::sync::mpsc::Receiver` based on **floem's own** `crossbeam` feature. Enabling `crossbeam` ONLY on `floem_renderer` makes floem fail to compile (the renderer's `new_pending` wants crossbeam, floem's caller passes std mpsc — "distinct types"). FIX: put `features=["crossbeam"]` on the `floem` dep (it propagates to floem_renderer via `crossbeam = ["dep:crossbeam","floem_renderer/crossbeam"]`). Our own code does NOT use crossbeam directly.

**Incremental (streaming) decode + render (2026-06-09):** The GUI now fills the spectrogram in left-to-right as the file decodes (Spek-like), instead of one appearance after full decode+FFT. Architecture:
- `audio.rs`: added a streaming `Decoder` — `audio::open(path) -> Decoder` (decodes 1st packet eagerly to learn authoritative rate/channels, buffered as `pending_first`); `dec.next_chunk() -> Result<Option<Vec<i32>>>` (one decoded packet's interleaved samples, skips corrupt, None at EOF); `dec.total_len()` = `track.num_frames * channels` if the container reports it (it does for WAV/FLAC/MP3/…). `read()` (CLI batch) is now a thin loop over `next_chunk`. Source of total length: symphonia `Track.num_frames: Option<u64>`.
- `spectrogram.rs`: `Channel` now OWNS its `Dft` (dropped the `<'a>` lifetime / `&'a mut Dft` — was a borrow only for rayon reuse; ownership needed so a struct can hold per-channel state). `make_channel` takes `Dft` by value. New `pub struct StreamProcessor`: `new(cfg, rate, channels, total_len)` fixes geometry UP FRONT (total length required — that's why we need `total_len`), then `push(&interleaved_chunk)` deinterleaves per channel and `flow`s them in PARALLEL (`channels.par_iter_mut()`), `finish()` drains all, `into_spectrogram()` yields the final exact `Spectrogram`. Getters: `cols_done()`, `x_size/rows/rate/step_size/block_steps/chans/cfg`, `channel_dbfs(ch)`. Correctness: SoX's `flow` is buffer-boundary-independent (buffers internally across calls), so chunked feeding == one big `flow` → streaming final pixels are byte-identical to batch (verified CLI `rs_*` references still IDENTICAL after the refactor).
- `render.rs`: factored `dbfs_to_rgba(cfg, dbfs, rows, cols_filled, total_cols, autogain)` — paints `0..cols_filled` from dbfs, leaves the rest as background palette[0] (so the image grows L→R at full target width while streaming). `channel_rgba` now calls it (cols_filled==total==spec.cols). New `stream_channel_images(proc)` renders each channel at width=`x_size`, filled=`cols_done`, autogain=0 (normalize OFF for GUI — global max unknown mid-stream).
- `gui.rs`: a worker thread (`load_async`→`worker`) decodes→accumulates ~100ms batches (`batch = rate*chans/10`)→`proc.push`→publishes a throttled (`PUBLISH_INTERVAL=60ms`, ~16fps, only when `cols_done` advanced) `LoadMsg::Frame(stream_channel_images)`; at EOF `finish()`+`into_spectrogram()`→`LoadMsg::Done(channel_images)` (exact). `LoadMsg::Meta` is sent FIRST (cols=x_size = full duration) so axes draw immediately before any pixels. Unknown-length containers fall back to decode-all + single `build_image` render (no progressive). The CLI `--gui file` preload now ALSO streams (calls `load_async`, not synchronous analyze).
  - UI marshalling: hand-rolled `make_sink(handler) -> Arc<dyn Fn(LoadMsg)+Send+Sync>` using `floem::ext_event::{ExtSendTrigger, register_ext_trigger}` + a `Scope::new()` `create_effect` that drains an `Arc<Mutex<VecDeque<LoadMsg>>>` on `trigger.track()`. Why not the built-ins: `create_ext_action` is FnOnce (one-shot); `update_signal_from_channel` funnels through a single-slot `Option` signal that can coalesce away intermediate values (would risk dropping Meta/Done). Our queue drains ALL → never drops. The effect runs on the UI thread (safe to `signal.set`); the worker thread calls `queue.push_back` + `register_ext_trigger`.
  - GOTCHA: `Arc<dyn Fn>` is NOT itself callable (std impls `Fn` for `&F`/`&mut F`/`Box<F>`, not `Arc`). Pass it to the worker as `&dyn Fn(LoadMsg)` (`let emit: &dyn Fn(LoadMsg) = &*sink;`).
